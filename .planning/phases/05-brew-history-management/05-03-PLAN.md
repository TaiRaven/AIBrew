Add after handleEditSave, verbatim from PATTERNS.md §Hard DELETE pattern:

```typescript
const handleDelete = async (sysId: string) => {
  const g_ck = (window as any).g_ck
  if (!g_ck) { setDeleteError('Session token not available.'); return }
  if (!SYS_ID_RE.test(sysId)) { setDeleteError('Invalid record identifier.'); return }
  try {
    const res = await fetch(`/api/now/table/${BREW_LOG_TABLE}/${sysId}`, {
      method: 'DELETE',
      headers: { 'X-UserToken': g_ck },
      // No Content-Type — DELETE has no body
      // No Accept — 204 No Content has no body
    })
    if (!res.ok) throw new Error(`HTTP ${res.status}`)
    // 204 No Content — do NOT call res.json() — it throws SyntaxError on empty body
    setDeleteTargetSysId(null)
    setEditBrew(null)
    setListKey(k => k + 1)
  } catch {
    setDeleteError("Couldn't delete brew — try again.")
  }
}
```

| Hard DELETE → brew_log record | Permanent deletion; no soft-delete (no active column on brew_log); confirmation modal is the only guard |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-09 | Tampering | handleDelete DELETE URL | mitigate | `SYS_ID_RE.test(sysId)` guard (`/^[0-9a-f]{32}$/i`) before URL construction — same pattern as PATCH; rejects injected/malformed sysId values |
| T-05-10 | Spoofing | handleDelete DELETE request | mitigate | `'X-UserToken': g_ck` header on DELETE request; g_ck checked, error returned if missing |
| T-05-11 | Tampering | 204 No Content response handling | mitigate | res.json() NOT called on DELETE response — prevents SyntaxError masking success as failure (RESEARCH.md Pitfall 2); only `res.ok` is checked |

| T-05-13 | Tampering | Race condition — two delete entry points | mitigate | Card trash icon calls setEditBrew(null) before setDeleteTargetSysId, ensuring only one confirmation modal can be open at a time (RESEARCH.md Pitfall 6) |
